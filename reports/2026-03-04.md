NIST finalized its first three post-quantum cryptography (PQC) standards in August 2024 — **ML-KEM** (CRYSTALS-Kyber), **ML-DSA** (CRYSTALS-Dilithium), and **SLH-DSA** (SPHINCS+). In 2026, the first wave of **binding PQC compliance requirements** from regulatory bodies (CISA guidance, EU Cyber Resilience Act, financial sector regulators) is expected to drop. The practical threat is already active: state actors are conducting **"harvest now, decrypt later" attacks** — recording encrypted TLS traffic today to decrypt when quantum computers powerful enough become available. NIST's IR 8547 explicitly names this attack class. Hashicorp published their PQC migration timeline in January 2026: "first PQC-compatible TLS certificates won't be commercially available until 2026." The Certificate Authorities (DigiCert, Let's Encrypt, Sectigo) are beginning to ship PQC hybrid certificates — combining traditional RSA/ECDSA with PQC algorithms in the same cert so legacy clients still work.

**So what:** This is a 2026–2031 migration window, not a fire drill today — but there are four things worth doing now. (1) **Audit our TLS stack**: OpenClaw gateway (port 7443) uses Node.js TLS → check `node -e "console.log(process.versions.openssl)"` (covered in Edition 13). (2) **Vercel, Convex, and Clerk** will handle PQC certificate migration on their end — no action from us required for production HTTPS. (3) **The harvest-now-decrypt-later threat is most relevant to long-lived sensitive data**: field tech invoice PDFs, Supabase permit records, and Clerk user credentials are the data we'd care about being decrypted in 5 years. Ensure these are stored encrypted at rest now (Supabase encrypts at rest by default; Convex does too). ## 2. 💳 x402 — HTTP 402 Finally Gets Its Moment: AI Agents Pay for API Calls in USDC Stablecoins
**Source:** DEV Community (`dev.to`) / chainstack.com / solana.com / bim.finance blog · **Published:** February–March 2026
**Relevance:** ⭐⭐⭐⭐ (4/5) — **Infrastructure story: the missing payment layer for autonomous AI agents doing paid API calls**
**Tags:** `#AI` `#agents` `#payments` `#x402` `#HTTP` `#USDC` `#crypto` `#infrastructure` `#coinbase`

**x402** is an open payment protocol built on the **HTTP 402 status code** — the "Payment Required" response code reserved in HTTP/1.0 in 1991 but never formally implemented. Developed by the **Coinbase Developer Platform team** and released as an open standard, x402 enables any API or web service to require payment before serving content, with AI agents as the primary intended client. Flow: an AI agent sends a request → server responds with HTTP 402 + payment details → agent sends a USDC micropayment on-chain → server validates and serves content → all in a single round-trip. Key design decisions: **USDC stablecoins** (no FX risk, no oracle), **sub-cent micropayments** viable (0.001 USDC per API call), **atomic transactions** (payment and content delivery can't be split), transparent audit trail on-chain. The DEV Community writeup (published 3 days ago) calls it "the standard competition will crystallize between 2026–2027" and recommends "integrating x402 support in Q2–Q3 2026 positions you ahead of the curve."

**So what:** x402 directly addresses the "how do AI agents pay for paid API services without a human in the loop?" problem — the same problem Agihalo on Product Hunt is attacking with their "autonomous x402/USDC recharging" feature.
