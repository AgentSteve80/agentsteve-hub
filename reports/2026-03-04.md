## 1. 🦆 Goose by Block — Jack Dorsey's Open-Source AI Agent, Provider-Agnostic, MCP-Extensible
**Source:** GitHub (`block/goose`) / r/LocalLLaMA / block.github.io · **Published:** Trending today March 4
**Relevance:** ⭐⭐⭐⭐ (4/5) — **Direct: provider-agnostic OpenClaw alternative built on MCP — comparison shapes our architecture choices**
**Tags:** `#AI` `#agents` `#open-source` `#MCP` `#LLM` `#developer-tools` `#comparison`

**Goose** (`block/goose`) is an open-source, extensible AI agent built by **Block** (Jack Dorsey's company — makers of CashApp, Square, and Spiral). The positioning: "goes beyond code suggestions — install, execute, edit, and test with **any LLM**." Goose is architecturally provider-agnostic by design: it connects to any LLM through a unified interface and extends via **MCP (Model Context Protocol) servers** — any existing MCP server works as a Goose toolkit without modification. Key differentiator from OpenClaw: Goose has no default model provider, no persistent conversation gateway, and no built-in memory system — it's a pure task-execution agent. The YouTube video published today ("Goose — The Open Source AI Agent That Replaces Your Entire Dev Workflow") is pushing it up GitHub trending. aiforcode.io's OpenClaw vs Goose comparison calls Goose "more opinionated about task boundaries" while OpenClaw is "more opinionated about context continuity." Goose's MCP extensibility means the same MCP servers we use for OpenClaw (file access, git, browser) can run under Goose with zero modification.

**So what:** Three angles for our stack. (1) **Architecture insight**: Goose's "provider-agnostic via MCP" model is converging with what OpenClaw does via its plugin system. If Anthropic ever has an outage or pricing spike, Goose's design shows how to route to Gemini/Ollama with zero config change — a pattern we could replicate in how our OpenClaw crons are invoked. (2) **Task parallelism**: Goose is designed for isolated task execution (start, do the task, done) rather than a persistent conversation agent. For Mission Control's pipeline automation — "trigger this scraper now, commit results" — Goose might be a lighter-weight option than spinning up a full OpenClaw session. (3) **Security posture**: Block ships Goose without a persistent gateway/API endpoint, which closes the attack surface that The New Stack flagged for OpenClaw (Edition 16). Worth benchmarking both for different use cases rather than defaulting to OpenClaw for everything. GitHub: `block/goose`.

**Source:** The Hacker News / The Record from Recorded Future / news4hackers · **Published:** March 3, 2026
**Relevance:** ⭐⭐⭐ (3/5) — **Threat intel: state-sponsored APTs now shipping Rust malware; pattern for credential theft techniques**
**Tags:** `#security` `#APT` `#India` `#espionage` `#Rust` `#malware` `#keylogger` `#threat-intel`

The threat cluster **SloppyLemming** — assessed as India-linked by Arctic Wolf researchers — launched a campaign from January 2025 through January 2026 targeting government entities and critical infrastructure in **Pakistan, Bangladesh, and Sri Lanka**. Specific targets include Pakistani nuclear regulatory bodies, defense logistics organizations, and telecommunications infrastructure; plus Bangladeshi energy utilities and financial institutions. The attack chain uses two distinct malware families delivered in parallel: **BurrowShell** (custom backdoor, C2 over encrypted channels) and a **Rust-based keylogger** (the first confirmed Rust keylogger used in this APT cluster). The Rust keylogger is notable: compiled Rust binaries are notoriously difficult to reverse-engineer (memory safety features + no standard ABI make static analysis harder), and the language is increasingly favored by APT developers for exactly this reason. **So what:** Two relevant patterns even though this campaign targets South Asian governments (not our stack). (1) **Rust malware is here in production.** The Rust-based keylogger confirms that threat actors have moved past "Rust is too complex to use for offensive tooling" — 2026 will see more Rust-compiled malware. Our SecureClaw security scanner (`github.com/AgentSteve80/secureclaw`) should add a check for processes running Rust-compiled binaries in expected paths (unusual for standard system tools). (2) **Spear-phishing impersonating a financial institution** is the exact attack model for phishing Andrew via email impersonating Convex, Stripe, Clerk, or Vercel — all services we use. The BurrowShell/keylogger combination targeting credential material is the same playbook as StegaBin's npm .env stealer (Edition 15) at a different sophistication level. ## 3. 🔐 Post-Quantum Cryptography — First Binding Compliance Requirements Expected in 2026, "Harvest Now, Decrypt Later" Active
**Source:** The Hacker News / quantumxc.com / NIST IR 8547 / Hashicorp Blog · **Published:** February–March 2026
**Relevance:** ⭐⭐⭐⭐ (4/5) — **DIRECT: every HTTPS connection our stack makes could be retroactively decrypted; PQC TLS certificates arrive this year**
**Tags:** `#security` `#cryptography` `#PQC` `#NIST` `#TLS` `#compliance` `#2026-deadline`

NIST finalized its first three post-quantum cryptography (PQC) standards in August 2024 — **ML-KEM** (CRYSTALS-Kyber), **ML-DSA** (CRYSTALS-Dilithium), and **SLH-DSA** (SPHINCS+). In 2026, the first wave of **binding PQC compliance requirements** from regulatory bodies (CISA guidance, EU Cyber Resilience Act, financial sector regulators) is expected to drop. The practical threat is already active: state actors are conducting **"harvest now, decrypt later" attacks** — recording encrypted TLS traffic today to decrypt when quantum computers powerful enough become available. NIST's IR 8547 explicitly names this attack class. Hashicorp published their PQC migration timeline in January 2026: "first PQC-compatible TLS certificates won't be commercially available until 2026." The Certificate Authorities (DigiCert, Let's Encrypt, Sectigo) are beginning to ship PQC hybrid certificates — combining traditional RSA/ECDSA with PQC algorithms in the same cert so legacy clients still work.

**So what:** This is a 2026–2031 migration window, not a fire drill today — but there are four things worth doing now. (1) **Audit our TLS stack**: OpenClaw gateway (port 7443) uses Node.js TLS → check `node -e "console.log(process.versions.openssl)"` (covered in Edition 13). The same OpenSSL 3.4.1+ update that patches CVE-2025-15467 also includes early PQC support via the `oqsprovider`. (2) **Vercel, Convex, and Clerk** will handle PQC certificate migration on their end — no action from us required for production HTTPS. (3) **The harvest-now-decrypt-later threat is most relevant to long-lived sensitive data**: field tech invoice PDFs, Supabase permit records, and Clerk user credentials are the data we'd care about being decrypted in 5 years. Ensure these are stored encrypted at rest now (Supabase encrypts at rest by default; Convex does too). (4) **Watch for PQC TLS in Node.js 24** (expected H1 2026) — once Node.js ships ML-KEM hybrid support, upgrading our OpenClaw Node.js version will add PQC to all outbound TLS connections automatically.

## 4. 📄 libxml2 "Enterprise Edition" — Former Maintainer Forks Under AGPL, Lobsters Debate Erupts
**Source:** Lobsters (`lobste.rs/s/rlof2z`) / Codeberg (`nwellnhof/libxml2-ee`) · **Published:** March 2–3, 2026
**Relevance:** ⭐⭐⭐ (3/5) — **Ecosystem pattern: foundational library maintainer drama → licensing divergence → transitive dependency risk**
**Tags:** `#open-source` `#licensing` `#libxml2` `#AGPL` `#nodejs` `#python` `#ecosystem` `#Lobsters`

**Nick Wellnhof**, the former lead maintainer of **libxml2** (who was replaced by GNOME's maintainers in late 2024 after a governance dispute), published a fork called **libxml2-ee** ("Enterprise Edition") on Codeberg under **AGPL v3** — the most restrictive OSI-approved license, which requires any software that links to an AGPL library to also be released under AGPL. The original libxml2 remains under MIT license. The Lobsters thread (11 comments, trending) has the key exchange: "Enterprises do not want to use AGPL software" (top comment), with discussion noting that since the fork doesn't accept external contributions and upstream remains MIT, the AGPL fork will likely see zero enterprise adoption. However, the situation creates confusion: `libxml2-ee` on Codeberg looks official because it's from the former maintainer, and any developer who unknowingly uses it instead of the MIT upstream could create a GPL-contamination issue in a commercial product.

**So what:** libxml2 is a transitive dependency in a surprising number of Node.js packages and Python libraries — it underlies XML parsing in many tools. Practically: (1) **Run `npm ls libxml2` and `pip show libxml2 lxml` in our project directories** to confirm we're pulling from the MIT upstream, not from any fork. (2) **The AGPL confusion pattern is going to repeat** as maintainer burnout (Edition 15 OSS crisis) drives more developers to "relicense or walk" moves. `libxml2-ee` is a preview of what happens when well-meaning security patches get trapped in a forked AGPL library: the fix exists but enterprises can't legally use it. (3) For SiteSignal's scraper stack: `lxml` (Python XML parsing) is in our venv. Verify it's linked to the MIT libxml2, not the AGPL fork. `pip show lxml` → check the version, then confirm it's pulling from the GNOME-maintained MIT upstream.
