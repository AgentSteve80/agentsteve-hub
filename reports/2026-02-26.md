## 1. ⚠️ CVE-2026-26960 — node-tar Hardlink Escape: Arbitrary File Write in Every npm Install
**Source:** GitLab Advisory / Windows Forum / OffSeq Threat Radar · **Published:** Feb 25–26, 2026
**Relevance:** ⭐⭐⭐⭐⭐ (5/5)
**Tags:** `#security` `#nodejs` `#npm` `#supply-chain` `#patch-now` `#stack`

**CVE-2026-26960** is a high-severity path traversal flaw in **node-tar** (`isaacs/node-tar`) — the package used by npm, yarn, and pnpm to extract archive files during `npm install`. A crafted tar archive can create a hardlink inside the extraction directory pointing to a file *outside* the extraction root, allowing arbitrary file read and write as the extracting user. Affects versions ≤ 7.5.7. **Fixed in node-tar 7.5.8**. Every Node.js project that runs `npm install` on untrusted or CI-fetched packages is in scope.

**So what:** This hits every project in our stack — Mission Control, Pal Web, Pal Field App. A supply chain attacker who compromises a dependency upstream could include a malicious archive that writes files outside the project directory when installed. Run `npm audit` on all three projects now; ensure node-tar is at 7.5.8+. Also check GitHub Actions runners — if CI does `npm install`, a compromised transitive dep could write to the runner filesystem. Related: pair this with `npm audit --audit-level=critical` in all CI pipelines.

## 2. ⚠️ CVE-2026-27606 — Rollup Bundler Path Traversal: Affects Vite, Next.js Build Chains
**Source:** Dev.to / CVEReports / Vulners · **Published:** Feb 26, 2026
**Relevance:** ⭐⭐⭐⭐⭐ (5/5)
**Tags:** `#security` `#rollup` `#nextjs` `#vite` `#build-toolchain` `#patch-now` `#stack`

**CVE-2026-27606** is an arbitrary file write vulnerability in **Rollup** — the bundler that powers Vite, and transitively Next.js and Expo's web builds. Due to a fragile regex-based blacklist for output filename sanitization (instead of proper path normalization), attackers can bypass the sanitization and write bundle output files to arbitrary locations on the filesystem. Affects Rollup < 2.80.0, < 3.30.0, and < 4.59.0. Patched in the latest versions of each branch.

**So what:** Mission Control and Pal Web both run on Next.js which uses Rollup under the hood via Webpack/Turbopack. This is exploitable in CI environments where build output paths may be user-influenced (e.g., via env vars or config injection from a compromised upstream dep). Run `npm list rollup` in all projects; update to a patched version. Also check if Vercel's build environment pins Rollup versions — if so, confirm they've patched their build images.

## 2. 🚨 Chrome CVE-2026-2441 — CSS Use-After-Free Zero-Day, Actively Exploited
**Source:** The Hacker News / SecurityWeek / The Register / Malwarebytes · **Published:** Feb 19–20, 2026
**Relevance:** ⭐⭐⭐⭐⭐ (5/5)
**Tags:** `#security` `#browser` `#cve` `#zero-day` `#patch-now` `#rce`

**CVE-2026-2441** (CVSS 8.8) is Chrome's first actively exploited zero-day of 2026 — a use-after-free bug in Chrome's CSS parsing engine that allows a remote attacker to execute arbitrary code *inside* the browser sandbox via a specially crafted HTML page. No click required beyond visiting the page. Patched in **Chrome 145.0.7632.75/76** (Windows/Mac) and **144.0.7559.75** (Linux). Google acknowledged exploitation in the wild before the patch dropped.

**So what:** If any machine touching our stack (VM, dev laptop, CI runner) has Chrome below 145, it's vulnerable to drive-by RCE just from visiting a malicious page. This includes Andrew's Mac Mini M4 and any browser-based testing sessions. Immediate action: confirm Chrome is updated on all dev machines. Also: our pal-web admin portal renders in a browser — if a field tech or admin visits a phishing page in the same browser profile they use for Pal Web, a compromised browser can exfiltrate Clerk session tokens. Browser isolation (separate profile for admin work) is worth recommending.

## 3. ⚠️ Dell RecoverPoint CVE-2026-22769 — Hardcoded Creds Exploited by China-Nexus UNC6201 Since 2024
**Source:** The Hacker News / Cybersecurity Dive / Léargas Security / eSentire · **Published:** Feb 18–20, 2026
**Relevance:** ⭐⭐⭐ (3/5)
**Tags:** `#security` `#cve` `#cisa-kev` `#china` `#infrastructure` `#hardcoded-creds`

**CVE-2026-22769** is a maximum-severity (CVSS 10.0) hardcoded credential vulnerability in **Dell RecoverPoint for Virtual Machines** — a VM backup and replication product used across enterprise datacenters. An unauthenticated attacker can use the hardcoded default credential to gain root-level persistent access to the underlying host. CISA added it to the KEV catalog on **Feb 18, 2026**, but the kicker: Google Mandiant attributes active exploitation to **UNC6201** (China-nexus) going back to **mid-2024** — meaning this was a live zero-day for ~18 months before disclosure. Federal agencies have until March 11 to patch.

**So what:** We don't use Dell RecoverPoint — this doesn't directly hit our stack. But the meta-lesson is critical: (1) hardcoded credentials in infrastructure software is still a top-tier attack vector in 2026, and (2) nation-state actors hold zero-days for months or years before detection. Audit any shared infrastructure for hardcoded defaults. Also: this is the kind of story the Mission Control Security page should surface — it validates our advisory feed design.

## 4. 💀 Grist-Core CVE-2026-24002 "Cellbreak" — Spreadsheet Formula Triggers Server RCE
**Source:** The Hacker News / Infosecurity Magazine / SecurityOnline · **Published:** Jan–Feb 2026
**Relevance:** ⭐⭐⭐ (3/5)
**Tags:** `#security` `#cve` `#self-hosted` `#rce` `#open-source` `#sandbox-escape`

**CVE-2026-24002** (CVSS 9.1), dubbed **"Cellbreak"** by Cyera Research Labs, is a critical Pyodide sandbox escape in **Grist-Core** — the popular open-source relational spreadsheet-database used by self-hosters as a lightweight Airtable/Notion alternative. A single malicious Python formula in a spreadsheet cell escapes the Pyodide WebAssembly sandbox and achieves RCE on the host server. If Grist is deployed as multi-tenant SaaS, it becomes RCE in the control plane affecting all tenants. Grist patched promptly; update immediately.

**So what:** If anyone on the Pal Low Voltage team is using self-hosted Grist for job tracking, parts inventory, or scheduling — this is a drop-everything-and-patch situation. More broadly, this illustrates the risk of "code-in-cells" architectures: Airtable formulas, Notion formulas, spreadsheet automation — all run untrusted user code. Any time you let users write "formulas," treat it as a code execution surface. Relevant pattern for future Pal features if we ever add formula-based job cost calculations.
