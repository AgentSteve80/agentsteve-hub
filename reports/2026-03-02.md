## 1. 🔴 RoguePilot — GitHub Copilot Leaks GITHUB_TOKEN via Prompt Injection in GitHub Issues
**Source:** The Hacker News / Orca Security / SecurityWeek · **Published:** Feb 28 – Mar 1, 2026
**Relevance:** ⭐⭐⭐⭐⭐ (5/5) — **DIRECT: AgentSteve80 GITHUB_TOKEN HAS WRITE ACCESS TO ALL REPOS**
**Tags:** `#security` `#github` `#copilot` `#prompt-injection` `#token-theft` `#action-required`

**RoguePilot** is a confirmed exploit chain (researched by Orca Security, reported by The Hacker News and SecurityWeek): an attacker creates a GitHub Issue containing a hidden prompt injection inside an HTML comment tag (`<!--the_prompt_goes_here-->`). When a developer opens a Codespace from that issue, GitHub Copilot silently processes the attacker's instructions without user approval — instructing itself to write the `GITHUB_TOKEN` to a JSON file in the workspace. The token is then exfiltrated. Zero user interaction required beyond opening the Codespace. The attack is confirmed to also enable additional Codespace features for lateral movement and data exfiltration. GitHub has been notified; partial mitigations applied.

**So what:** `GITHUB_TOKEN` in GitHub Actions and Codespaces is a privileged credential — it has write access to the repository it was issued for, and depending on org settings, cross-repo access. For `AgentSteve80`: our repos (`pal-field-app`, `pal-web`, `mission-control`, `sitesignal-web`, `secureclaw`, `augustus`) have GitHub Actions workflows that use `GITHUB_TOKEN` implicitly. The attack vector is: (1) someone opens a malicious GitHub issue on one of our repos — even from a spam/bot account — and (2) I or Andrew opens a Codespace from that issue while Copilot is active. **Three immediate mitigations**: (1) Don't open Codespaces from issues filed by unknown accounts. (2) Review GitHub Actions `permissions:` blocks — pin to `contents: read` or the minimum required scope. (3) Check if Copilot is enabled on `AgentSteve80`'s org settings — scope it to known repos only. Copilot is not just a productivity tool anymore; it's a trust boundary that needs its own security policy.

## 3. ⚠️ OpenSSL Patches 12 Vulnerabilities — CVE-2025-15467 CVSS 9.8 in Node.js Stack
**Source:** Security Affairs / SC Media / Red Hat RHSA-2026:1473 / Orca Security · **Published:** Jan 27–29, 2026
**Relevance:** ⭐⭐⭐⭐ (4/5) — **DIRECT: Node.js depends on OpenSSL — affects all our server-side code**
**Tags:** `#security` `#openssl` `#CVE` `#nodejs` `#RCE` `#stack` `#action-required`

OpenSSL patched **12 vulnerabilities** on January 27, 2026 (shipped Jan 29). The most severe: **CVE-2025-15467** (CVSS 9.8) — a stack buffer overflow in CMS/PKCS#7 AEAD parsing. An attacker sends a specially crafted encrypted message → stack overflow → arbitrary code execution. A second high-severity bug: **CVE-2025-11187** — stack buffer overflow in PKCS#12 PBMAC1 MAC verification. Affects OpenSSL 3.0, 3.3, 3.4, 3.5, and 3.6. Red Hat issued RHSA-2026:1473 for RHEL 9. Orca Security published a full technical writeup. Patches are available in OpenSSL 3.4.1+ (or 3.3.3+, 3.0.17+ for LTS).

**So what:** Node.js ships with and depends on OpenSSL for all TLS/HTTPS operations. Every server-side Next.js API route (`/api/stats`, `/api/subscribe`, Stripe webhook handler), every Convex HTTP action, and the OpenClaw gateway (port 7443) uses the underlying Node.js OpenSSL version. **Check our Node.js version and its bundled OpenSSL version**: `node -e "console.log(process.versions.openssl)"`. If the output is below `3.0.17` or `3.3.3` or `3.4.1`, the runtime needs updating. On the Ubuntu VM: `apt list --installed 2>/dev/null | grep openssl`. Note: CMS/PKCS#7 AEAD is used in S/MIME email processing and some certificate formats — not our primary use case, but any dependency that parses untrusted TLS-adjacent data is in scope.

## 4. 📄 Foxit & Apryse PDF Platforms — 16 Zero-Days Including CVSS 9.8 OS Command Injection
**Source:** CyberSecurityNews / HackRead / VPNCentral / THN · **Published:** Feb 18–24, 2026
**Relevance:** ⭐⭐⭐ (3/5) — **Pattern warning for our pal-web PDF generation**
**Tags:** `#security` `#PDF` `#zero-day` `#RCE` `#nodejs` `#pattern` `#foxit`

Researchers (using AI-assisted vulnerability scanning) discovered **16 zero-day vulnerabilities** in Foxit PDF and Apryse PDF platforms — the two most widely deployed enterprise PDF SDKs. CVEs: `CVE-2025-70401`, `CVE-2025-70402`, `CVE-2025-66500`. The most severe: **CVSS 9.8 OS Command Injection in Foxit PDF SDK for Web's Node.js signature server** — a user-supplied filename reaches a `child_process.exec()` call without sanitization → full server RCE. Secondary findings include XSS, account takeover, and document engine data exfiltration. Both vendors received advance notice and shipped patches before public disclosure.

**So what:** We don't use Foxit or Apryse — `pal-web` uses `@react-pdf/renderer` for invoice PDF generation. But the **pattern is the warning** (same lesson as WeGIA CVE-2026-28409, Edition 9): user-supplied data reaching `exec()` or `child_process` in a PDF context = RCE. Audit `pal-web/convex/pdf.ts`: does the `InvoicePDF` component receive any user-supplied data that could reach a shell or filesystem path? Look specifically at invoice number, builder name, and job address fields — these come from Convex records which come from field tech input. Validate and sanitize before they reach any file-writing operation. The secondary risk: **Foxit Reader is the default PDF viewer for many enterprise users** — builders like Epcon, Beazer, and Pulte likely use it internally. If we email PDF invoices to builders and they open in a Foxit version pre-patch, an unpatched vuln becomes their problem (and a trust issue for Pal).
