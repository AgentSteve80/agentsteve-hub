### 1. üî• Infostealer Malware Now Actively Targeting OpenClaw Config Files & Gateway Tokens
**Relevance: 10/10** `#security` `#openclaw` `#infostealer` `#immediate`
**Links:** [The Hacker News](https://thehackernews.com/2026/02/infostealer-steals-openclaw-ai-agent.html) ¬∑ [Security Affairs](https://securityaffairs.com/188097/malware/hackers-steal-openclaw-configuration-in-emerging-ai-agent-threat.html) ¬∑ [CyberInsider](https://cyberinsider.com/infostealer-malware-now-targeting-openclaw-ai-environments/) ¬∑ [eSecurity Planet](https://www.esecurityplanet.com/threats/infostealers-target-openclaw-ai-configuration-files/)

Infostealers are now specifically capturing OpenClaw AI agent environments. The stolen file set includes: **`openclaw.json`** (gateway authentication token, email, workspace path), **`device.json`** (private cryptographic keys), and **soul/memory markdown files** (agent behavioral config and personal context). This wasn't a targeted OpenClaw module ‚Äî a broad file-grabbing routine looking for sensitive extensions and directory names accidentally scooped up the entire OpenClaw environment. Multiple victims confirmed.

**So what:** This is the most directly relevant story in weeks. Three actions right now: **(1)** Your `~/.openclaw` directory is the exact target. Ensure your Mac doesn't have any infostealers ‚Äî run `malwarebytes` or equivalent on the Mac Mini host. **(2)** The gateway token in `openclaw.json` is what lets an attacker control your OpenClaw instance remotely ‚Äî rotate your gateway token after ensuring the Mac is clean. **(3)** The "soul and memory files" being leaked means an attacker would get your SOUL.md, MEMORY.md, and daily notes ‚Äî everything about how you use OpenClaw, your projects, your credentials references. This is a strong argument for keeping MEMORY.md minimal and not logging actual secret values there.

### 2. üî¥ Starkiller Phishing-as-a-Service ‚Äî Bypasses MFA by Proxying the Real Login Page
**Relevance: 9/10** `#security` `#phishing` `#mfa` `#pal-field-app`
**Links:** [Krebs on Security](https://krebsonsecurity.com/2026/02/starkiller-phishing-service-proxies-real-login-pages-mfa/) ¬∑ [Dark Reading](https://www.darkreading.com/threat-intelligence/starkiller-phishing-kit-mfa) ¬∑ [Infosecurity Magazine](https://www.infosecurity-magazine.com/news/starkiller-phishing-kit-bypasses/)

**Starkiller** is a new commercial phishing-as-a-service platform that defeats MFA by acting as a transparent proxy in front of the real login page. When a victim is phished, they're authenticating against the *actual* Clerk/Google/Microsoft site through Starkiller's reverse proxy ‚Äî the phishing service captures their session token in real time. The GUI makes it trivially accessible to low-skill attackers: URL masking, session hijacking, container lifecycle management, all automated. Brian Krebs broke the story.

**So what:** Standard TOTP or SMS MFA doesn't protect against this ‚Äî the attacker captures the live session. The only protection that works is **hardware security keys** (FIDO2/passkeys), which are phishing-resistant because they verify the origin domain. For Pal Field App specifically: Clerk supports passkeys. When Pal goes live with real users, consider strongly encouraging (or requiring) passkey enrollment over TOTP. For your own accounts: your Anthropic, GitHub, Convex, and Vercel accounts should all use hardware key MFA if you have a YubiKey.

### 3. üî¥ MCP Servers Have a Critical RCE Flaw ‚Äî 10,000+ Servers Potentially Affected
**Relevance: 8/10** `#security` `#mcp` `#rce` `#ai`
**Links:** [CyberPress](https://cyberpress.org/mcp-servers-expose-data-risk/) ¬∑ [Medium ‚Äî MCP State of the Ecosystem](https://medium.com/@amilasilvapvt/how-the-model-context-protocol-is-about-to-reshape-the-way-humanity-buys-sells-and-lives-104be36ddef3)

A critical remote code execution vulnerability was disclosed in the Model Context Protocol (MCP) server implementation. MCP has exploded to **10,000+ published servers** as the de-facto standard for connecting AI models to external tools (Slack, databases, APIs). The flaw allows a malicious MCP server to execute arbitrary code in the client that connects to it ‚Äî meaning if you connect OpenClaw or Claude to a compromised or malicious MCP server, your machine is owned.

**So what:** You're not running MCP servers currently, so direct risk is low. But watch the MCP server ecosystem carefully before adding any. If you ever wire Mission Control or Pal Web to Claude via MCP tools, make sure those MCP servers are self-hosted (not third-party). The 10,000-server number with essentially zero security review is a supply chain attack waiting to happen.

### 4. üî¥ 20 AI App Data Breaches Since 2025 ‚Äî All the Same Three Root Causes
**Relevance: 8/10** `#security` `#saas` `#pal-field-app` `#pal-web`
**Links:** [Barrack.ai](https://blog.barrack.ai/every-ai-app-data-breach-2025-2026/)

Barrack.ai catalogued every documented AI application security incident since January 2025 ‚Äî 20 total, tens of millions of users exposed. The root causes are identical across every breach: **(1) missing or misconfigured authentication**, **(2) API keys hardcoded, committed, or left in public-facing configs**, **(3) misconfigured user permissions (any user can access any data)**.

**So what:** These are exactly the failure modes to audit in Pal before launch. Specific checklist: **(1)** Convex row-level access rules ‚Äî does every query verify the caller's identity? **(2)** Are your Convex and Clerk keys anywhere in the repo (check with `git log -S "your_key_string"`)?  **(3)** Can a tech see another tech's invoices or mileage records? Each of these is a GDPR/liability issue for a real company. Run Claude Code Security scan (from last night's report) across all three repos this week.
