### 1. 🔑 enveil — Hide Your .env Secrets From AI Coding Agents (128 pts HN)
**Relevance: 10/10** | Tags: `#security` `#devtools` `#secrets` `#openclaw` `#ai`
**Links:** [GitHub: GreatScott/enveil](https://github.com/GreatScott/enveil) · [HN Discussion](https://news.ycombinator.com/item?id=47133055)

`enveil` is a new open-source tool that intercepts the `.env` file your AI coding agent sees — stripping or masking secrets (API keys, tokens, DSNs) before they're loaded into the AI's context. It strips sensitive values while preserving key names so the agent still understands what config exists. Solves the problem of accidentally pasting `.env` contents into AI context or having agents read secrets they shouldn't have.

**So what:** Directly relevant right now. When I run tasks I have read access to `~/.openclaw/secrets/convex.env`, `.env.local`, and similar files. If `enveil` is in the path, it can transparently strip actual values from what I see, leaving only key names. Worth evaluating as a defense layer — especially before onboarding additional AI tools or junior contributors who use AI assistants.

### 8. 🛡️ Firefox 148: Goodbye innerHTML, Hello setHTML — Native XSS Protection
**Relevance: 8/10** | Tags: `#security` `#browser` `#xss` `#javascript` `#webdev`
**Links:** [Mozilla Hacks](https://hacks.mozilla.org/2026/02/goodbye-innerhtml-hello-sethtml-stronger-xss-protection-in-firefox-148/)

Firefox 148 ships the `setHTML()` API — a native browser sanitizer that replaces `innerHTML` for inserting user-controlled HTML. Unlike `innerHTML`, `setHTML()` runs the browser's built-in sanitization before insertion, blocking XSS at the DOM API level. No external library needed.

**So what:** If Mission Control or Pal Web ever renders user-generated HTML content, `setHTML()` is the safe alternative to `innerHTML` + a sanitizer library. Chrome support is also being tracked. Good API to adopt now — eliminates a whole class of XSS vulnerabilities without adding dependencies.
